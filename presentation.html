Data Preparation & Chunking Strategy

Raw tabular data is converted into 3 types of text documents - this is the key design decision that makes analytical queries work.

1. Transaction Descriptions - 5,009 docs

Each order grouped into a natural language paragraph with items, totals, profit margin, customer info, and location.

Order CA-2016-152156 placed on Nov 08, 2016 by Claire Gute (Consumer) from Henderson, KY (South region). Shipped via Second Class. Items: Bush Bookcase $261.96, Hon Chairs $731.94 Order total: $993.90, profit margin: 26.3%

2. Aggregated Summaries - 141 docs

Pre-computed rollups: 48 monthly, 16 quarterly, 4 yearly, 20 category/subcategory, 53 regional/state summaries with year-over-year growth percentages.

3. Statistical Summaries - 18 docs

Top/bottom products, discount impact analysis, seasonality patterns, segment breakdowns, region comparisons, shipping mode analysis.

Why This Approach?

❌ Naive approach: Convert each row to text, split by character count. The LLM gets random transaction fragments - cannot answer "what's the yearly sales trend?"

✅ Our approach: Semantic chunking + pre-computed aggregations. Each chunk is a complete logical unit. Summaries contain the aggregated data the LLM needs for analytical questions.

All chunks stay well within the MiniLM embedding model's 256-token sweet spot. No arbitrary character splitting - each document is a semantically complete unit.

RAG Pipeline - Retrieval & Generation

Step 1: Query Classification

Each question is classified by keywords into categories:

trend category regional comparative profitability seasonality

Classification determines which extra document types to boost during retrieval.

Step 2: Hybrid Retrieval

Combines general similarity search with targeted summary searches:

  • Trend query → boost yearly + monthly + quarterly summaries
  • Category query → boost category + subcategory summaries
  • Regional query → boost regional + state summaries
  • Results deduplicated and ranked by cosine similarity

Step 3: Prompt Engineering

System prompt constrains the LLM to:

  • Use only provided data
  • Be specific with numbers
  • Flag insufficient data
  • Temperature: 0.1 for factual accuracy

Why Hybrid Search?

Simple similarity search for "sales trend over 4 years" returns individual transactions mentioning "sales." Hybrid search ensures we also pull yearly summaries and trend statistics - the documents that actually answer the question.

Discussion of Results

Key Findings from the Data

📈 Growth Trend

Revenue grew 51% from 2014 to 2017 ($484K → $733K), with the strongest growth in 2016 (+29.5%). The business is on a clear upward trajectory.

🪑 Furniture Problem

Furniture generates $742K in sales but only $18K profit (2.5% margin). Heavy discounting in sub-categories like Tables (negative profit) is the root cause. Biggest optimization opportunity.

🌍 Regional Imbalance

West region (especially California) dominates. Central region has the worst profit margin (7.9%) despite decent volume - suggesting pricing or cost issues in that region.

📅 Seasonality

Strong Q4 seasonality across all years. November–December consistently peak. This pattern is valuable for inventory planning and marketing timing.

Answer Quality Assessment

Answers consistently reference specific dollar amounts, percentages, and rankings from the source data. The LLM rarely hallucinates numbers because the context contains pre-computed summaries with exact figures.

Limitations & Future Work

Current Limitations

⚠️ No Real-Time Computation

The system retrieves pre-computed summaries. It cannot compute new aggregations on the fly (e.g., "average discount for orders over $500 in Q3 2016"). Answers are limited to what was pre-aggregated during data preparation.

⚠️ Small Model Limitations

Mistral 7B occasionally misinterprets complex multi-part questions or produces less detailed answers compared to larger models like Mistral 7B or GPT-4. Trade-off for speed.

⚠️ Context Window Constraint

4,000 character context limit means some complex queries may not receive all relevant data. Very broad questions ("tell me everything about 2017") may get incomplete context.

⚠️ Text-Only Responses

Answers are text-only. Charts and graphs would significantly improve the analytical experience, especially for trend and comparison questions.

Potential Improvements

🔧 SQL Agent Integration

Add a SQL tool so the LLM can query the raw data directly for ad-hoc computations, combining RAG retrieval with live data access for arbitrary questions.

🔧 Larger Model Option

Upgrading to a better model would improve answer quality for complex analytical questions, at the cost of slower inference (~50s vs ~15s).

🔧 Chart Generation

Have the LLM generate matplotlib or plotly code for visual answers - trend lines, bar charts, pie charts alongside text responses for richer analysis.

🔧 Evaluation Framework

Implement automated evaluation with ground-truth answers to systematically measure retrieval precision, answer accuracy, and hallucination rate.
